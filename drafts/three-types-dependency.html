I've been thinking a lot about modules.

Dependency is directional

A common approach to thinking about modularity is the notion of "coupling". You may have seen the diagram below before (public domain):

I have an issue with this diagram I think coupling is directional. That's because coupling is ultimately about dependency.

We like to think of coupling between modules as "interaction" between two things. But this seems wrong to me. Contemporary methods for writing code are hierarchical.

For procedural or object-oriented programming, you enter a program at one place, then execution flows from place to place in a thread of execution. For functional programming, you have a main function which is expanded recursively into one big evaluation. In both cases, you start at the top, and every procedure call or function reference brings you down one level, and it all eventually returns and control is thrown back up the chain.

But the point is, when foo() { bar() }, foo depends on bar, but bar does not depend on foo. The case where they depend on each other is what we call recursion, and it's a special case, not the norm -- even so, it's just made up of two function calls, each of which is a directional dependency.

Three layers of dependency

A dependency of a piece of code is something that that piece of code requires in order to run successfully. But that is a poor definition, as it just pushes the word "depend" into "require". Good enough for informal discussions, but not good enough for mathematics.
